Event Search & Analytics

Kinesis Event Search With Athena

This document explains the end-to-end pattern implemented for moving Kinesis messages through Amazon Data Firehose into S3, partitioning the files by message type, enriching records with Kinesis metadata, crawling the raw archive with Glue, and exposing a clean Athena view for table-style analysis.

Why This Solution Was Needed

Developers and support teams often need to answer practical investigation questions: did a specific message arrive, what was inside it, which Kinesis shard and sequence number carried it, and did it reach S3 correctly? Looking directly in Kinesis can help for immediate troubleshooting, but it is not a comfortable or durable investigation interface.

Challenge Why It Happens Impact On The Team
Short retention window Kinesis stores records only for the configured retention period. Older messages may no longer be available when an incident or reconciliation request arrives.
Shard-oriented access Records are distributed across shards based on the partition key. Investigators need to know which shard to inspect, or they must scan across shards manually.
Sequence number dependency Precise retrieval often depends on knowing the Kinesis sequence number or an iterator position. If the sequence number was not captured elsewhere, finding the exact message is slow and uncertain.
No SQL-style search Kinesis does not provide native filtering by fields inside the JSON payload. Searching by customer id, account number, brand, status, or message type requires custom code or manual inspection.
Encoded payloads Kinesis and Firehose Lambda events represent record data as bytes or base64-encoded strings. Teams must decode the payload before they can confirm the business content.
Nested and stringified JSON The business payload is inside events[0].data as a JSON string. Even after retrieval, the team still has to parse inner JSON before useful fields are visible.
Operational access limits Not every analyst or support engineer should have direct stream read permissions. Debugging becomes dependent on a smaller group of AWS operators or developers.
Manual correlation Kinesis metadata and payload data are separate unless explicitly stored together. Teams may struggle to prove which S3 object, Athena row, or downstream record came from which Kinesis message.

Typical Support Questions

A typical investigation question is not "show me the next record in shard 3." It is usually closer to "find the Customers message for account 10770331-TEVA that arrived around this time" or "confirm whether sequence number 496685... was delivered to S3." Kinesis alone does not make those questions easy to answer because it is optimized for streaming consumption, not ad hoc analytics.

Design decision: use S3 as the durable message archive and Athena as the searchable support interface, while preserving Kinesis shard, partition key, sequence number, and arrival timestamp on every queryable row.

Athena As The Saviour

Athena gives developers and support teams a familiar SQL interface over the raw archive. Instead of reading shards, decoding records, and manually parsing nested strings, teams can query a saved view and filter by business fields or Kinesis metadata.

  • Longer retention: S3 becomes the durable raw archive, independent of Kinesis retention.
  • SQL access: Athena can filter by account number, brand, customer id, partition key, or sequence number.
  • Line-level metadata: each flattened row carries the Kinesis shard id, sequence number, and arrival timestamp.
  • Lower operational friction: support and data teams can query Athena instead of directly reading Kinesis shards.
  • Auditability: S3 keeps the original message structure, while Athena views make the data easier to inspect.

Executive Summary

The working solution keeps the original producer message shape, writes raw records to S3 through Firehose, extracts events[0].partitionKey for S3 folder routing, enriches each record with Firehose and Kinesis metadata through a Firehose Lambda transformation, and uses Athena views to flatten the nested arrays into queryable customer rows.

Current outcome: records land under folders such as partition_id=Customers/, Athena can query the raw Glue table, and a saved view returns one row per customer account with the Kinesis sequence number, shard id, partition key, and arrival timestamp repeated on every row.

Architecture Flow

  1. Producer: sends a Kinesis record whose payload contains an events array.
  2. Kinesis Data Stream: stores the event and assigns shard, sequence, and arrival metadata.
  3. Amazon Data Firehose: reads from the Kinesis stream and applies dynamic partitioning.
  4. Firehose Lambda transformation: enriches the original payload with available Kinesis metadata.
  5. Amazon S3 raw archive: stores newline-delimited raw JSON files under partitioned prefixes.
  6. AWS Glue Crawler: creates or updates a raw Glue Data Catalog table.
  7. Amazon Athena: queries the raw table and exposes a reusable flattened view.
Producer
  -> Kinesis Data Stream
  -> Data Firehose
  -> Lambda transform adds metadata
  -> S3 raw archive partition_id=Customers/yyyy/MM/dd/
  -> Glue raw table
  -> Athena customer_flat_v view

Source Message Shape

The producer sends a wrapper JSON object. The important routing field is events[0].partitionKey. The business payload is inside events[0].data as a JSON string. 

{
  "events": [
    {
      "data": "{ \"party\": [ ... customer payload ... ] }",
      "partitionKey": "Customers"
    }
  ]
}

This shape is valid for routing and raw archival, but Glue sees data as a string. That is why Athena uses json_parse(e.data) inside the saved view to expose normal columns.

Why Dynamic Partitioning Was Configured This Way

Firehose dynamic partitioning was needed so records of different message types land in separate S3 folders. Since the partition key is present in the payload as events[0].partitionKey, Firehose can extract it with inline JSON parsing.

Firehose Field Configured Value Purpose
Dynamic partitioning Enabled Routes records to S3 prefixes based on extracted metadata.
Inline parsing for JSON Enabled Allows Firehose to run JQ expressions against the source record.
Key name partitionKey Name used by the S3 prefix token.
JQ expression .events[0].partitionKey Extracts values such as Customers.
S3 bucket prefix partition_id=!{partitionKeyFromQuery:partitionKey}/!{timestamp:yyyy}/!{timestamp:MM}/!{timestamp:dd}/ Creates folders by message type and date.
Error prefix errors/!{firehose:error-output-type}/!{timestamp:yyyy}/!{timestamp:MM}/!{timestamp:dd}/ Separates failed delivery, processing, or parsing records.

Why Newline Delimited JSON Matters

Athena JSON SerDe expects one complete JSON object per line. During testing, pretty-printed multi-line JSON caused read errors because Athena tried to parse individual lines instead of full records. The fix was to send compact single-line JSON to Kinesis and enable the Firehose newline delimiter setting. 

{"events":[{"data":"{\"party\":[...]}","partitionKey":"Customers"}]}
{"events":[{"data":"{\"party\":[...]}","partitionKey":"Customers"}]}
Operational note: if multiple Firehose records are written to the same S3 object without a newline delimiter, Athena can see glued JSON objects and fail to parse them.

Lambda Metadata Enrichment

Firehose can use Kinesis metadata for delivery processing, but it does not automatically write that metadata into S3. A Firehose Lambda transformation was added so metadata becomes part of the delivered JSON record and can be queried later.

Metadata Captured

  • firehose_record_id for Firehose delivery traceability.
  • firehose_arrival_timestamp for Firehose processing time.
  • kinesis_shard_id to identify the source shard.
  • kinesis_partition_key to confirm routing value.
  • kinesis_sequence_number to trace back to Kinesis Data Viewer.
  • kinesis_subsequence_number for aggregated-record scenarios.
  • kinesis_arrival_timestamp for when Kinesis received the message.

Delivered S3 Record Shape

{
  "events": [
    {
      "data": "{ \"party\": [ ... ] }",
      "partitionKey": "Customers"
    }
  ],
  "_firehose_metadata": {
    "firehose_record_id": "shardId-000000000003...",
    "kinesis_shard_id": "shardId-000000000003",
    "kinesis_partition_key": "Customers",
    "kinesis_sequence_number": "49668599872568290257233820367181262314268963292089155634",
    "kinesis_arrival_timestamp": 1781762871480
  }
}

Glue Crawler Design

Glue is used to create the raw table metadata for Athena. Because different partition keys can represent very different JSON schemas, each message type should have its own crawler or stable S3 source path.

Setting Recommendation Reason
S3 source path s3://om-stream-kinesis-archive/partition_id=Customers/ Restricts the crawler to one logical dataset.
Grouping behavior Create a single schema for each S3 path Prevents one table per Firehose output file.
Subsequent runs Crawl all sub-folders during testing Safest while schema and files are being validated.
Schema change policy Update table definition Allows new top-level fields such as _firehose_metadata.
Important: test files with different structures caused Glue to create many file-named tables. The fix was to remove old inconsistent test files and keep only files with the final stable structure.

Athena View For Line-Level Querying

The raw Glue table shows the wrapper JSON as nested arrays and a string payload. A saved Athena view flattens events, parses data, unnests party and customers, and attaches Kinesis metadata to every resulting row. 

CREATE OR REPLACE VIEW customer_flat_v AS
WITH parsed AS (
  SELECT
    _firehose_metadata,
    e.partitionkey,
    CAST(
      json_parse(e.data) AS ROW(
        party ARRAY(
          ROW(
            party_identifier VARCHAR,
            customer_name VARCHAR,
            account_type_code VARCHAR,
            account_type_name VARCHAR,
            status VARCHAR,
            parent_number VARCHAR,
            customers ARRAY(
              ROW(
                customer_identifier VARCHAR,
                party_identifier VARCHAR,
                account_number VARCHAR,
                secondary_customer_name VARCHAR,
                status VARCHAR,
                account_status_name VARCHAR,
                acct_class_id VARCHAR,
                acct_class_name VARCHAR,
                collector_id VARCHAR,
                collector_name VARCHAR,
                cust_freight_term_code VARCHAR,
                cust_freight_term_name VARCHAR,
                ucc128_label_format VARCHAR,
                apply_past_cancel_hold VARCHAR,
                zero_freight_customer VARCHAR,
                group_invoices_by_bill_to VARCHAR,
                cust_freight_account VARCHAR,
                sales_channel_code VARCHAR,
                sales_channel_name VARCHAR,
                demand_class_code VARCHAR,
                distributor_channel VARCHAR,
                brand VARCHAR,
                ship_method VARCHAR
              )
            )
          )
        )
      )
    ) AS payload
  FROM customerpartition_id_customers t
  CROSS JOIN UNNEST(t.events) AS x(e)
)
SELECT
  partitionkey,
  _firehose_metadata.kinesis_shard_id,
  _firehose_metadata.kinesis_partition_key,
  _firehose_metadata.kinesis_sequence_number,
  from_unixtime(_firehose_metadata.kinesis_arrival_timestamp / 1000) AS kinesis_arrival_time,
  p.party_identifier,
  p.customer_name,
  p.status AS party_status,
  c.customer_identifier,
  c.account_number,
  c.secondary_customer_name,
  c.status AS customer_status,
  c.brand,
  c.ship_method
FROM parsed
CROSS JOIN UNNEST(payload.party) AS x(p)
CROSS JOIN UNNEST(p.customers) AS y(c);

Benefits Of The Implemented Pattern

Routing

Clean S3 Folder Separation

Dynamic partitioning sends each message type to a predictable folder such as partition_id=Customers/, which keeps schemas isolated.

Traceability

Kinesis Metadata In S3

Sequence number, shard id, partition key, and arrival timestamp can be queried in Athena and matched back to Kinesis Data Viewer.

Usability

Reusable Athena View

Users do not need to manually parse nested JSON every time. They query customer_flat_v like a normal table.

Operations

Raw Archive Preserved

The original message remains available in S3 for replay, audit, troubleshooting, and future transformation into Parquet.

Performance Roadmap

The current view works well for validation and operational lookups. For high-volume reporting across thousands of files, the next step is to build a curated Parquet layer. Parquet reduces scanned data, improves compression, avoids repeated JSON parsing, and performs better when combined with file compaction. 

Raw S3 JSON archive
  -> Glue ETL or Athena CTAS
  -> Parse events[0].data
  -> Flatten customer rows
  -> Write compacted Parquet
  -> Athena curated table

Target Parquet file sizes should generally be in the 128 MB to 512 MB range. This avoids the small-files problem caused by thousands of tiny Firehose output objects.

Operational Checklist

  • Send compact single-line JSON to Kinesis.
  • Keep Firehose newline delimiter enabled.
  • Keep dynamic partition key name and S3 prefix token aligned.
  • Run the Glue crawler again only when the outer schema or partitions change.
  • Keep test files with incompatible shapes out of the production crawler path.
  • Query the saved Athena view for day-to-day validation and traceability.
  • Plan a Parquet compaction layer before broad production reporting.
